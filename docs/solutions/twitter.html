Design Twitter Timeline and Search

Difficulty Hard
Concepts
Fan-out Timeline Search CQRS Message queues Cache

Step 1: Outline use cases and constraints

Gather requirements and scope the problem. Ask questions to clarify use cases and constraints. Discuss assumptions.

Use cases

We'll scope the problem to handle only the following use cases

  • User posts a tweet
    • Service pushes tweets to followers, sending push notifications and emails
  • User views the user timeline (activity from the user)
  • User views the home timeline (activity from people the user is following)
  • User searches keywords
  • Service has high availability

Out of scope

  • Service pushes tweets to the Twitter Firehose and other streams
  • Service strips out tweets based on users' visibility settings
  • Analytics

Constraints and assumptions

General

  • Traffic is not evenly distributed
  • Posting a tweet should be fast
  • 100 million active users
  • 500 million tweets per day or 15 billion tweets per month
  • 250 billion read requests per month
  • 10 billion searches per month

Timeline

  • Viewing the timeline should be fast
  • Twitter is more read heavy than write heavy
  • Ingesting tweets is write heavy

Search

  • Searching should be fast
  • Search is read-heavy

Calculate usage

  • Size per tweet: ~10 KB average (including media)
  • 150 TB of new tweet content per month
  • 100 thousand read requests per second
  • 6,000 tweets per second
  • 60 thousand tweets delivered on fanout per second
  • 4,000 search requests per second

Step 2: Create a high level design

Outline a high level design with all important components.

Step 3: Design core components

Use case: User posts a tweet

Delivering tweets and building the home timeline is trickier. Fanning out tweets to all followers will overload a traditional relational database. We'll want to choose a data store with fast writes such as a NoSQL database or Memory Cache.

  • The Client posts a tweet to the Web Server
  • The Write API stores the tweet in the user's timeline on a SQL database
  • The Fan Out Service queries the User Graph Service to find followers
  • Stores the tweet in the home timeline of followers in a Memory Cache
  • Stores the tweet in the Search Index Service
  • Uses the Notification Service to send push notifications

Use case: User views the home timeline

  • The Read API contacts the Timeline Service
  • Gets timeline data from Memory Cache
  • Queries Tweet Info Service for tweet details
  • Queries User Info Service for user details

Use case: User searches keywords

  • The Search API contacts the Search Service
  • Parses/tokenizes the input query
  • Queries the Search Cluster (Lucene)
  • Scatter gathers results, merges, ranks, and returns

Step 4: Scale the design

Identify and address bottlenecks, given the constraints.

The Fanout Service is a potential bottleneck. Users with millions of followers could take several minutes to have their tweets go through the fanout process.

Optimizations:

  • Keep only several hundred tweets for each home timeline in the Memory Cache
  • Keep only active users' home timeline info in Memory Cache
  • Store only a month of tweets in Tweet Info Service
  • Store only active users in User Info Service
